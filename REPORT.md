## Раздел 1: Описание проблемы

### Что такое Race Condition?

_TODO: Объясните, что такое race condition в контексте баз данных._

Race condition (состояние гонки) — это ситуация, когда в контексте баз данных несколько транзакций одновременно пытаются прочитать и изменить одни и те же данные.

**Пример из жизни:**
- Два клиента одновременно пытаются купить последний товар на складе. Оба проверяют наличие (SELECT count), оба видят «1 шт.», оба создают заказ и уменьшают остаток (UPDATE count = count - 1). В итоге остаток становится -1, а товара на самом деле не хватило.

### Почему READ COMMITTED не защищает от двойной оплаты?

_TODO: Объясните механизм работы READ COMMITTED и почему он не предотвращает race condition._

На уровне изоляции READ COMMITTED:
1. Каждая транзакция видит только те данные, которые были зафиксированы (committed) до начала текущего запроса, а не всей транзакции.
2. Когда две транзакции одновременно проверяют статус заказа (например, status = 'created'), они обе видят его актуальным, так как никто еще не успел закоммитить изменения.
3. В результате обе транзакции проходят проверку бизнес-логики и отправляют запрос на UPDATE. Первая транзакция блокирует строку, вторая ждет. После COMMIT первой транзакции, вторая «просыпается», делает свой UPDATE поверх и повторно выполняет действие (например, списывает деньги).

**Демонстрация проблемы:**

```
Время | Сессия 1                    | Сессия 2
------|----------------------------|---------------------------
t1    | BEGIN                      |
t2    | SELECT status (created)    |
t3    |                            | BEGIN
t4    |                            | SELECT status (created)
t5    | pg_sleep(3)                |
t6    |                            | pg_sleep(3)
t7    | UPDATE status = paid       |
t8    |                            | UPDATE status = paid (ждет)
t9    | COMMIT                     |
t10   |                            | UPDATE выполняется...
t11   |                            | COMMIT
```

_TODO: Объясните, что происходит на каждом этапе._
Объяснение этапов:
t1, t3: Начало транзакций.
t2, t4: Обе сессии видят статус created. Условие «заказ еще не оплачен» верно для обеих.
t5, t6: Ожидание.
t7: Сессия 1 обновляет статус и вешает эксклюзивную блокировку на строку.
t8: Сессия 2 пытается обновить ту же строку, но встает в очередь, пока Сессия 1 не завершится.
t9: Коммит Сессии 1.
t10: После коммита Сессии 1, Сессия 2 успешно перезаписывает статус на paid (хотя он и так уже был paid), выполняя логику оплаты во второй раз.

### READ COMMITTED (по умолчанию)

**Описание:**

Уровень, используемый в Postgres по умолчанию. Гарантирует, что запрос увидит только зафиксированные данные.

**Пример non-repeatable read:**
```sql

SELECT balance FROM accounts WHERE id = 1; -- Результат: 1000
-- Сессия 2 изменяет balance и делает COMMIT
SELECT balance FROM accounts WHERE id = 1; -- Результат: 500 (!)
COMMIT;
```

**Когда использовать:**

- Для большинства обычных операций: чтение каталога, просмотр профилей, некритичные обновления. Подходит, когда производительность важнее строгой согласованности, а приложение само обрабатывает конфликты.

## Раздел 3: Решение проблемы

### Почему REPEATABLE READ решает проблему?

_TODO: Объясните, как REPEATABLE READ помогает решить проблему конкурентных оплат._

REPEATABLE READ использует snapshot isolation:
1. Транзакция видит данные на момент своего старта.
2. Это означает, что если вторая сессия изменила статус заказа, первая сессия об этом «не узнает» из своего SELECT.
3. Однако в PostgreSQL на этом уровне действует правило: если транзакция пытается обновить строку, которая была изменена другой закоммиченной транзакцией после создания снимка первой, первая транзакция завершится с ошибкой: ERROR: could not serialize access due to concurrent update. Это заставляет приложение обработать ошибку и не допустить двойной оплаты.

### Зачем нужен FOR UPDATE?

_TODO: Объясните роль FOR UPDATE в решении проблемы._

`FOR UPDATE` создает эксклюзивную блокировку на уровне строки (row-level lock):

**Без FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...';
-- Другая транзакция может прочитать ту же строку
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**С FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...' FOR UPDATE;
-- Другая транзакция ЖДЕТ освобождения блокировки
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**Типы блокировок:**
- `FOR UPDATE` — эксклюзивная блокировка (никто не может изменить или заблокировать)
- `FOR SHARE` — разделяемая блокировка (другие могут читать с FOR SHARE, но не могут изменять)
- `FOR NO KEY UPDATE` — как FOR UPDATE, но разрешает concurrent FOR KEY SHARE
- `FOR KEY SHARE` — слабая блокировка (предотвращает DELETE и UPDATE ключевых полей)

### Что произойдет без FOR UPDATE на REPEATABLE READ?

_TODO: Опишите сценарий, когда REPEATABLE READ без FOR UPDATE не работает._

Даже на REPEATABLE READ возможна аномалия:

```sql
-- Сессия 1
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
COMMIT;

-- Сессия 2 (параллельно)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...'; -- created (snapshot!)
-- Задержка...
UPDATE orders SET status = 'paid' WHERE id = '...' AND status = 'created';
-- ЧТО ПРОИЗОЙДЕТ?
```

_TODO: Объясните, что произойдет в этом случае._

Без FOR UPDATE обе транзакции читают snapshot данных на момент своего старта и видят статус `created`. Поскольку REPEATABLE READ в PostgreSQL при попытке обновить строку, которая была изменена другой закоммиченной транзакцией после создания snapshot, выбрасывает ошибку `ERROR: could not serialize access due to concurrent update`, первая закоммиченная транзакция пройдёт, а вторая получит ошибку и будет откачена. Однако без FOR UPDATE нет гарантии, что вторая транзакция «увидит» изменение первой до момента UPDATE — она может попытаться обновить строку, которая уже была изменена, что приведёт к откату. На практике PostgreSQL корректно обнаружит конфликт, но блокировка FOR UPDATE делает это поведение детерминированным: вторая транзакция ждёт первую, а затем видит уже обновлённый статус и корректно отклоняется на уровне бизнес-логики.

### Разница между FOR UPDATE и FOR SHARE

| Характеристика | FOR UPDATE | FOR SHARE |
|----------------|------------|-----------|
| Тип блокировки | Эксклюзивная | Разделяемая |
| Блокирует чтение с FOR UPDATE | ✅ | ✅ |
| Блокирует чтение с FOR SHARE | ✅ | ❌ |
| Блокирует UPDATE | ✅ | ✅ |
| Блокирует DELETE | ✅ | ✅ |
| Use case | Перед изменением | Защита от изменений |

**Пример использования FOR SHARE:**
```sql
-- Проверка наличия товара перед созданием заказа
SELECT quantity FROM products WHERE id = '...' FOR SHARE;
-- Товар не может быть удален или изменен до COMMIT
```

## Раздел 4: Рекомендации для продакшена

### Какой ISOLATION LEVEL использовать для продакшена маркетплейса?

_TODO: Дайте обоснованную рекомендацию по выбору уровня изоляции для продакшена._

**Рекомендация:** Для продакшена маркетплейса рекомендуется использовать **READ COMMITTED как default уровень** с явным использованием **REPEATABLE READ + FOR UPDATE для критичных операций**.

#### Обоснование:

**1. Производительность**
- READ COMMITTED имеет минимальный overhead
- REPEATABLE READ используется только там, где необходимо
- Избегаем глобального использования SERIALIZABLE

_TODO: Расширьте обоснование._

READ COMMITTED — уровень по умолчанию в PostgreSQL, оптимизированный для высокой пропускной способности. Если поднять изоляцию до SERIALIZABLE на всей базе, каждая конфликтующая транзакция может получить serialization failure, что потребует retry-логики и существенно снизит throughput. Гибридный подход позволяет использовать SERIALIZABLE только там, где это критично, а остальную нагрузку обрабатывать с минимальными накладными расходами.

**2. Безопасность данных**
- Критичные операции (оплата, изменение баланса) защищены через REPEATABLE READ + FOR UPDATE
- Некритичные операции (просмотр каталога) работают на READ COMMITTED
- Четкое разделение зон ответственности

_TODO: Расширьте обоснование._

Такой подход гарантирует, что финансовые операции (списание средств, изменение баланса) никогда не будут выполнены дважды благодаря блокировке FOR UPDATE и более высокому уровню изоляции. При этом обычные операции чтения не страдают от накладных расходов блокировок, что важно для отзывчивости интерфейса.

**3. Риски deadlock**
- FOR UPDATE может привести к deadlock при неправильном порядке блокировок
- Решение: всегда блокировать ресурсы в одном порядке
- Пример: сначала order, потом order_items, потом payment

_TODO: Расширьте обоснование._

Deadlock возникает, когда две транзакции взаимно блокируют друг друга, каждая ждёт ресурс, занятый другой. Если в приложении ресурсы захватываются в разном порядке, вероятность deadlock растёт экспоненциально. Строгое правило порядка блокировок (например, всегда сначала заказы, затем товары, затем платежи) полностью исключает deadlock. PostgreSQL автоматически обнаруживает deadlock и откатывает одну из транзакций, но это дополнительная нагрузка и необходимость retry.

**4. Простота разработки**
- READ COMMITTED легко понять и отлаживать
- REPEATABLE READ + FOR UPDATE требует explicit intent
- Код становится самодокументируемым

_TODO: Расширьте обоснование._

Когда разработчик явно указывает `FOR UPDATE` в критичных запросах, это служит документацией: любой, читающий код, сразу понимает, что здесь важна атомарность операции. Нет «магии» глобальных настроек — изоляция применяется точечно. Это упрощает код-ревью и снижает количество ошибок.

**5. Масштабируемость**
- READ COMMITTED хорошо масштабируется
- Локальное использование блокировок не создает bottleneck
- Connection pooling работает эффективно

_TODO: Расширьте обоснование._

При READ COMMITTED соединения быстро освобождаются, так как нет долгих блокировок. Пул соединений (например, PgBouncer) работает оптимально. Блокировки FOR UPDATE удерживаются минимальное время, что не создаёт очередей из ожидающих транзакций даже при горизонтальном масштабировании (несколько инстансов приложения).

### Альтернативные подходы

#### Подход 1: Использовать SERIALIZABLE везде

**Плюсы:**
- Максимальная корректность
- Не нужно думать о блокировках
- PostgreSQL автоматически обнаруживает конфликты

**Минусы:**
- Значительное снижение производительности (20-50%)
- Требует retry logic во всех операциях
- Высокий процент rollback при нагрузке
- Сложная отладка serialization failures

**Вывод:** Не рекомендуется для high-load систем.

#### Подход 2: Optimistic Locking (версионирование)

**Реализация:**
```sql
ALTER TABLE orders ADD COLUMN version INTEGER DEFAULT 1;

UPDATE orders 
SET status = 'paid', version = version + 1
WHERE id = '...' AND status = 'created' AND version = 1;

-- Проверить ROW_COUNT, если 0 — конфликт
```

**Плюсы:**
- Нет блокировок на чтение
- Хорошая производительность
- Масштабируется горизонтально

**Минусы:**
- Требует изменения схемы БД
- Требует retry logic в приложении
- Может быть много конфликтов при высокой нагрузке

**Вывод:** Хороший подход для распределенных систем.

#### Подход 3: Advisory Locks

**Реализация:**
```sql
BEGIN;
SELECT pg_advisory_xact_lock(hashtext('order_' || order_id));
-- Критическая секция
COMMIT; -- Блокировка автоматически снимается
```

**Плюсы:**
- Гибкий контроль блокировок
- Можно блокировать логические ресурсы
- Работает на любом уровне изоляции

**Минусы:**
- Легко забыть снять блокировку
- Сложнее отлаживать
- Требует дисциплины от разработчиков

**Вывод:** Полезно для специфичных случаев, но не как основной подход.

### Итоговая рекомендация

Для продакшена маркетплейса используйте **гибридный подход**:

1. **Default isolation level:** `READ COMMITTED`
   - Для обычных операций (чтение каталога, просмотр истории)
   - 95% операций

2. **REPEATABLE READ + FOR UPDATE:** для критичных операций
   - Оплата заказа
   - Изменение баланса
   - Резервирование товара
   - 5% операций

3. **Дополнительно:** Optimistic locking для операций с высокой конкурентностью
   - Обновление счетчиков
   - Изменение рейтингов

**Пример кода:**
```python
async def pay_order(order_id: UUID):
    async with db.transaction(isolation='repeatable_read'):
        # Заблокировать заказ
        order = await db.fetch_one(
            "SELECT * FROM orders WHERE id = $1 FOR UPDATE",
            order_id
        )
        
        if order['status'] != 'created':
            raise OrderAlreadyPaidError()
        
        # Обновить статус
        await db.execute(
            "UPDATE orders SET status = 'paid' WHERE id = $1",
            order_id
        )
        
        # Записать в историю
        await db.execute(
            "INSERT INTO order_status_history (order_id, status) VALUES ($1, 'paid')",
            order_id
        )
```

- Гибридный подход (READ COMMITTED по умолчанию + REPEATABLE READ + FOR UPDATE для критичных операций) оптимален для продакшена
